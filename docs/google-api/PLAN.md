# ggsc (GooGle-SearCh) — 方案规划

> **背景：** 基于 Cloudflare Workers 的 Google 搜索解析方案已被证明不可行（详见 [`docs/google-cf/ISSUES.md`](../google-cf/ISSUES.md)）。
> 本文档记录自建方案的规划与演进历程。

---

## 1. 核心需求

- 每天数万次 Google 搜索请求
- 结构化返回搜索结果（标题、URL、摘要）
- 自动处理 CAPTCHA
- 代理高可用、自动故障转移

---

## 2. 方案演进

### Phase 1 — MongoDB 代理池方案（已废弃）

最初设计基于 MongoDB 的动态代理池：
- 从公开源采集免费 SOCKS 代理
- 两级健康检查（TCP → Google 可达性）
- MongoDB 存储代理状态和黑名单

**废弃原因：**
- 免费代理质量差，成功率低于 5%
- 代理池维护复杂，投入产出比低
- MongoDB 依赖增加部署复杂度

### Phase 2 — WARP SOCKS5 + 备用代理（已废弃）

使用 Cloudflare WARP 作为主代理：
- socks5://127.0.0.1:11000 (WARP) 作为主力
- http://127.0.0.1:11111 和 11119 作为备用

**废弃原因：**
- Google 对 WARP 出口 IP 风控严格，触发 CAPTCHA 频率极高
- 主/备切换逻辑增加复杂度

### Phase 3 — 固定 HTTP 代理 + 负载均衡（当前方案）

最终架构：
- 两个本地 HTTP 代理（11111 + 11119）地位平等
- Round-robin 轮换负载均衡
- 健康检查 + 自动故障转移
- VLM 自动 CAPTCHA 识别

---

## 3. 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 浏览器自动化 | Playwright + 反检测策略 | 轻量、无 chromedriver 依赖 |
| HTTP 代理 | 本地 HTTP 代理 (11111, 11119) | 稳定、可控 |
| API 框架 | FastAPI | 异步、自动文档、类型安全 |
| HTML 解析 | BeautifulSoup4 | 灵活、容错 |
| CAPTCHA | VLM API | 自动识别，无需人工 |
| CLI | argparse | Python 标准库，无额外依赖 |
